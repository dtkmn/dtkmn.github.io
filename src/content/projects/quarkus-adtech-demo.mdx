## What it is

This project frames a more useful engineering question than a simple runtime shootout: when the
workload is bursty, latency sensitive, and horizontally scaled, which part of the request path is
actually dominating the result?

## Why it matters

Most runtime comparisons are too abstract to be useful. They compare hello-world startup times or
microbenchmarks with no operational story behind them. Here the scenario is concrete: a bid receiver
for Real-Time Bidding traffic, with HTTP-only and Kafka-backed modes, observability, and deployment
paths that look like a real system instead of a slide.

## What makes it credible

The project is interesting because it does not rely on a single flattering number. It brings together:

- multiple benchmark modes,
- side-by-side implementations,
- explicit workload and resource controls,
- load-test setup,
- observability tooling,
- a Helm deployment path.

That is the kind of material that lets an article make a defensible argument instead of a rhetorical one.

## How it connects to the writing

The original companion article on Quarkus native versus Go is now best read as the first snapshot.
The newer writing on what microservice benchmarks actually measure reflects where the project has
grown: a broader benchmark harness whose main value is not a winner table, but a clearer way to
separate framework cost from end-to-end bottlenecks.
